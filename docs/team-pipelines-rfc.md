# RFC: Declarative Agent Team Pipelines

**Status:** Design  
**Target version:** v0.3  
**Scope:** Claude Code plugin (`/recommend` skill + new `/team` flow); VS Code extension `/team` slash command

---

## Problem

Discovering and installing individual agents is solved. The next gap: teams of agents need coordination — someone has to break the task down, assign scoped work, and prevent two agents from editing the same file simultaneously. Today developers are left to wire this up manually.

---

## APM Capability Audit (v0.9.4)

`apm.yml` is a **flat dependency manifest**. It supports:

```yaml
name: my-project
version: 1.0.0
dependencies:
  apm:            # agent/skill/instruction packages
    - github/awesome-copilot/agents/foo.agent.md
  mcp:            # MCP server packages
    - name: my-mcp
      command: node
      args: ["dist/server.js"]
```

It has **no native support** for: hierarchy, stages, roles, handoffs, dependency graphs between agents, or execution ordering. `apm install` downloads files and pins a lockfile — it does not register agents with any runtime orchestration.

**Conclusion:** The team layer is ours to design. Our schema must be additive — `team:` and `agents:` keys are unknown to APM so it ignores them, but our tool reads them for orchestration hints. APM continues to read only `dependencies.apm` for installs.

## UX Flow

### Trigger

Regular search results already appear after `@agent-discovery <query>`. When the result set covers more than one domain (detected: ≥2 distinct agent types or ≥3 results), the response appends a "Build a team?" button.

Users can also invoke it directly: `@agent-discovery /team <query>`.

```
User: @agent-discovery building a react project

[search results: React Expert, TypeScript Dev, Grumpy Reviewer, Test Writer, Azure Deployment]

> These 5 agents cover 3 domains (frontend, quality, devops). Build a coordinated team?

[Build a team]  [Install individually]
```

### Acceptance path

Clicking **"Build a team"** triggers the `agent-discovery.buildTeam` command:

1. Groups results into hierarchy layers (see below).
2. Shows a markdown preview of the team structure in the chat.
3. Presents two buttons: **"Save team"** / **"Edit in file"**.
4. **Save team**: writes `.claude/teams/<slugified-query>.yml` and updates `apm.yml` (adds individual agent deps).
5. **Edit in file**: opens the team file in the VS Code editor before committing.

### Decline path

Clicking **"Install individually"** falls through to the current `/install` flow, running `installEntry()` for each result.

## Hierarchy Layers

Three-tier model maps cleanly to Claude Code's subagent architecture:

| Tier | Role | Responsibility |
|---|---|---|
| **Orchestrator** | One agent at the top | Breaks task into sub-tasks, assigns work, resolves conflicts |
| **Specialists** | 1–4 domain agents | Scoped implementation — each owns a file-path glob |
| **Quality gates** | 1–2 review/test agents | Run after specialists, gate progression to next stage |

A fourth optional tier for devops/release agents runs last.

**Layer assignment heuristic** (based on catalog entry tags/name):
- `orchestrator / lead / manager / planner` → Orchestrator tier
- `frontend / backend / typescript / python / [language]` → Specialist tier
- `review / grumpy / audit / security / test / qa` → Quality tier
- `deploy / devops / azure / ci / cd / release` → Release tier
- Anything else → Specialist tier

If no orchestrator candidate is found in the catalog, a generic "Project Lead" stub is injected.

## File Format

Files live at `.claude/teams/<name>.yml`.

**Concrete example — "building a react project":**

```yaml
# Agent team generated by Agent Discovery v0.3
# APM reads only `dependencies.apm` — the `team:` and `agents:` keys are
# ignored by APM but read by Claude Code for orchestration context.

name: react-project-team
version: 1.0.0
description: Agentic team for building a React project

team:
  orchestrator: react-project-lead
  stages:
    - name: planning
      agents: [react-project-lead]
    - name: implementation
      agents: [expert-react-frontend-engineer, typescript-developer]
      parallel: true
    - name: quality
      agents: [grumpy-reviewer, test-writer]
      requires: implementation
    - name: release
      agents: [azure-deployment]
      requires: quality

agents:
  react-project-lead:
    source: github/awesome-copilot/agents/react-project-lead.agent.md
    role: orchestrator
    scope: "**/*"
  expert-react-frontend-engineer:
    source: github/awesome-copilot/agents/expert-react-frontend-engineer.agent.md
    role: specialist
    scope: "src/components/**,src/pages/**,src/styles/**"
  typescript-developer:
    source: github/awesome-copilot/agents/typescript-developer.agent.md
    role: specialist
    scope: "src/**/*.ts,src/**/*.tsx,*.config.ts"
  grumpy-reviewer:
    source: github/awesome-copilot/agents/grumpy-reviewer.agent.md
    role: reviewer
    scope: "**/*"
  test-writer:
    source: github/awesome-copilot/agents/test-writer.agent.md
    role: quality
    scope: "src/**/*.test.ts,src/**/*.spec.tsx"
  azure-deployment:
    source: github/awesome-copilot/agents/azure-deployment.agent.md
    role: devops
    scope: ".github/**,azure/**,Dockerfile,*.bicep"

# APM-compatible flat dependency list — `apm install` reads this
dependencies:
  apm:
    - github/awesome-copilot/agents/react-project-lead.agent.md
    - github/awesome-copilot/agents/expert-react-frontend-engineer.agent.md
    - github/awesome-copilot/agents/typescript-developer.agent.md
    - github/awesome-copilot/agents/grumpy-reviewer.agent.md
    - github/awesome-copilot/agents/test-writer.agent.md
    - github/awesome-copilot/agents/azure-deployment.agent.md
```

## Guardrails: What "Don't Step Over Each Other" Means in Practice

Three mechanisms, in priority order:

### 1. Scope boundaries (`scope` field)

Each agent declares a file-glob scope. The orchestrator enforces this before delegating: it reads the team YAML, maps sub-task to agent, and only dispatches to an agent whose scope covers the relevant files.

Example: a request to "add a CSS module" goes to `expert-react-frontend-engineer` (scope covers `src/styles/**`), not `typescript-developer`.

### 2. Stage sequencing (`requires` field)

`requires: implementation` in the `quality` stage means reviewers run only after all `implementation` agents have returned. This is a soft constraint today (the orchestrator respects it via its instructions); a future runtime could enforce it as a DAG.

### 3. Parallel flag (`parallel: true`)

Marks agents in the same stage as safe to run concurrently because their scopes are disjoint. The orchestrator can spawn both as subagents simultaneously without risk of conflict.

## What Claude Code Does With the Team File

On startup, Claude Code scans `.claude/teams/` and loads any `.yml` files found. The active team:

1. Is surfaced in `/agent-discovery:list` output under "Teams"
2. Has its `agents` entries' `source` paths downloaded on first use (if not already present via `apm install`)
3. Sets the orchestrator agent as the default subagent when `/team` is invoked

No changes to Claude Code internals are needed for v0.3 — the orchestrator agent's own instructions handle delegation by reading the team YAML from `.claude/teams/` at runtime.

## `apm.yml` Integration

The team file is self-contained and its `dependencies.apm` list is a superset of what `apm install` needs. The root `apm.yml` can optionally reference the team:

```yaml
# apm.yml (project root)
name: my-project
version: 1.0.0
dependencies:
  apm:
    - local:.claude/teams/react-project-team.yml   # APM resolves transitive deps
```

APM's transitive resolution (`apm.lock.yaml`) would then pull in all agents listed in the team file. This is the cleanest integration — one entry in `apm.yml` expands to the full team.

*Note: whether APM supports `local:` path references needs confirmation against v0.9.4 schema. Fallback: list agents directly in the root `apm.yml` as we do today.*

## Implementation Plan (v0.3)

### Claude Code plugin

1. **`skills/recommend/SKILL.md`** — add "Build a team?" branch after search results when ≥3 multi-domain results are returned. Generate and write team YAML via MCP `write_file` tool.
2. **`src/mcp-server.ts`** — add `build_team` tool: takes search results + query, returns structured team YAML string.
3. **`src/team-builder.ts`** — pure function: `buildTeamYml(query, results) → string`. Assigns tiers via tag heuristics, generates YAML, deduplicates `dependencies.apm`.

### VS Code extension

4. **`src/team.ts`** — port `buildTeamYml()` from Claude Code plugin.
5. **`src/extension.ts`** — add `agent-discovery.buildTeam` command; add "Build a team?" button to `formatResults()` when ≥3 results; add `/team` slash command handler.
6. **`package.json`** — register `/team` command in `chatParticipants.commands`.

### Estimated scope

~4 hours for both platforms. The tier-assignment heuristic and YAML generation are the interesting parts; the rest is wiring.
